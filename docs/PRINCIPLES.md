# 训练与推理原理简述

本文用本仓库的闲聊花花中文语音包为例，解释 GPT-SoVITS 的训练、推理和服务化部署中最容易混淆的几个概念。

## 整体思路

GPT-SoVITS 可以理解为两个模型协同工作：

1. **GPT / Text2Semantic**：把输入文字转换成接近语音节奏和内容的语义 token。
2. **SoVITS / VITS**：把语义 token、文本特征、参考声音特征等信息转换成最终波形。

训练时，本仓库把碎片化中文语音整理成 `音频路径|说话人|语言|文本`，再生成 HuBERT 特征、语义 token、说话人向量等中间文件。微调后的 GPT 权重学习“这类文本如何变成该角色风格的语义表达”，SoVITS 权重学习“如何把这些语义表达合成为接近该角色的声音”。

## 已经微调过模型，为什么推理还需要参考语音和参考文字？

在 GPT-SoVITS 里，微调权重和参考输入不是二选一，而是共同参与推理：

| 输入 | 作用 |
| --- | --- |
| 微调后的 GPT 权重 | 提供当前角色/数据集的文本到语义 token 生成能力 |
| 微调后的 SoVITS 权重 | 提供当前角色/数据集的声学合成能力 |
| 参考语音 | 提供本次推理要模仿的音色、韵律、音量、口吻等条件信息 |
| 参考文字 | 告诉模型参考语音里实际说了什么，帮助模型对齐“参考音频内容”和“参考音色/韵律” |
| 目标文字 | 本次真正要合成的文本 |

所以，即使模型已经针对闲聊花花微调，参考语音仍然相当于“本次发声风格的条件”。它会影响音色稳定性、语速、情绪、停顿和音量。参考文字则用于让模型正确理解参考音频，不把参考音频中的内容误当成目标文本，也减少音素/语义对齐错误。

实践上，参考语音建议满足：

1. 3-10 秒，太短信息不够，太长会拖慢推理或引入多余风格。
2. 背景干净，尽量没有音乐、音效或多人声。
3. 与目标输出期望风格接近，例如语气、音量、情绪接近。
4. 参考文字必须准确对应参考语音。

本仓库默认参考语音使用 `data\smbw_zh_train\audio\TWzh__TalkFlower_Placement_Stream__Course_051_00.mp3`，参考文字为 `只有被选中的人才能得到璀璨的光芒。`，因为它满足 GPT-SoVITS 对 3-10 秒参考音频的要求。

## 训练过程在做什么？

本仓库把训练固化成五个阶段：

1. **数据准备和清理**：解包音频、抽取中文、ASR 标注、过滤太短或不适合训练的片段。
2. **格式化训练集**：生成 GPT-SoVITS 需要的文本、HuBERT、语义 token、说话人向量等中间数据。
3. **训练 SoVITS**：学习如何把语义和条件信息合成为角色声音。
4. **训练 GPT**：学习中文文本到语义 token 的生成模式。
5. **测试模型**：用固定文本和固定参考音频生成 wav，检查音色、清晰度、速度和稳定性。

`data`、`logs`、`GPT_weights_v2ProPlus`、`SoVITS_weights_v2ProPlus`、`outputs` 都是重要训练资产，分别对应原始/清理数据、中间特征与日志、训练权重和推理测试结果。

## 大量文字 TTS 或 API service 怎么降低冷启动时间？

如果只是命令行脚本，每次执行都会重新启动 Python、加载 BERT、HuBERT、GPT、SoVITS 和权重文件，冷启动会很明显。做 API service 时应避免“一次请求启动一次模型”，而是让服务进程常驻。

建议：

1. **服务启动时加载模型**：启动 API 时初始化 `TTS(config_path)`，把模型常驻在 GPU/内存里。
2. **请求只做推理**：每个请求只传目标文字、参考音频/参考文字和采样参数，不重复加载权重。
3. **固定参考音频时缓存条件特征**：如果整个服务都使用同一个角色参考音频，可以进一步缓存参考音频相关特征，避免每次重复处理参考音频。当前脚本没有实现这层缓存，但服务化时值得做。
4. **复用进程和 CUDA context**：不要频繁创建/销毁 Python 进程，否则 CUDA 初始化和模型加载会成为主要耗时。
5. **按文本长度分段**：大量长文本应先切句，分段生成，再拼接或流式返回，避免单次请求过长导致延迟和显存压力上升。
6. **限制并发**：单张 GPU 上不要无控制并发推理。可以用请求队列或 worker 池，按显存和延迟目标控制并发数。
7. **预热**：服务启动后先跑一次短文本推理，提前完成 CUDA kernel 初始化、缓存构建和模型路径检查。
8. **区分冷启动和热推理指标**：README 中的测速脚本会在同一进程内做 warmup 和多次 runs，更接近服务常驻后的热推理速度。

本机实测中，CUDA 平均 RTF 约 0.531，CPU 平均 RTF 约 1.196。也就是说，GPU 热推理快于实时播放，CPU 则慢于实时播放；构建 API service 时应优先使用 GPU 常驻模型。

## API service 的最小结构

最小可行服务结构如下：

1. 服务启动：读取固定配置，加载 GPT/SoVITS/BERT/HuBERT。
2. 预热：用短文本生成一次音频。
3. 请求进入：校验文本长度、语言、采样参数。
4. 推理：调用常驻 `TTS.run(...)`。
5. 输出：返回 wav bytes、文件路径，或流式返回音频片段。

如果未来要在本仓库里实现 API，建议复用 `scripts\talkflower-tts.py` 里的模型加载和参数组织逻辑，但把 `TTS` 实例提升为服务级单例，不要放在每个请求函数内部创建。
