从 v1 到 v2ProPlus clean 再到 v2pp strict：闲聊花花中文 TTS 训练实录

这是一篇基于本仓库和历史会话复盘整理的工程博客：我们如何把一包碎片化中文游戏语音，一步一步清理、训练、试听、排查问题，最后固化成可复用的 GPT-SoVITS v2ProPlus 训练和验证流程。

项目：talking-flower-voice 模型：GPT-SoVITS v2ProPlus 最新版：talkflower_zh_v2pp_strict 平台：Windows + CUDA

目标：不只是训练一个角色语音包

这个仓库的直接目标，是训练一个任天堂游戏角色“闲聊花花”的中文 TTS 语音包。更通用地看，它也是一次“如何用碎片化中文语音素材训练 TTS 模型”的工程实践：原始素材短、散、目录结构复杂，有些语音只有一两个字，有些文件更像音效或语气声，直接训练很容易得到“能说话但不像、发虚、破音、吐字不清”的模型。

因此，我们没有把训练当成一次性命令，而是拆成可审计、可回退、可复用的阶段：环境准备、数据抽取、自动标注、数据清理、格式化、SoVITS 训练、GPT 训练、固定参考音频验证、GPU/CPU 速度测试。

本文只记录工程过程。复用到其他素材时，需要确认语音素材、模型权重和衍生产物具备相应使用或分发权限。

阶段 0：先把工程环境固定下来

一开始我们先把当前目录规范成 Git 仓库根目录，并把关键约束写进仓库：使用 PowerShell 脚本初始化虚拟环境，依赖安装走阿里云 PyPI mirror，PyTorch 使用 CUDA 12.8 版本，预训练模型优先从 ModelScope 下载，避免依赖 Hugging Face 的网络环境。 

Set-ExecutionPolicy -Scope Process Bypass
.\scripts\init-venv.ps1
.\scripts\download-modelscope-model.ps1 -DownloadChineseAsr

这一步产出的不是模型，而是“可重复的训练工作台”：.venv、requirements.txt、pyproject.toml、scripts\init-venv.ps1、scripts\download-modelscope-model.ps1，以及后续训练所需的 GPT-SoVITS 预训练权重、中文 BERT、CNHuBERT、G2PW 和中文 ASR 模型。

阶段 1：从原始 mp3 zip 中只抽中文素材

原始数据来自 smbw_voice.zip。我们只需要中文语音，所以第一轮数据准备只抽取 CNzh 和 TWzh 两个目录，并把音频整理到训练目录中，再生成 GPT-SoVITS 使用的 list 格式： 

wav_path|speaker|zh|text

当前仓库把这一步固化为： 

.\scripts\stage-01-prepare-data.ps1 `
  -ZipPath "C:\Users\richard\Downloads\smbw_voice.zip" `
  -OutputRoot ".\data\smbw_zh_train"

早期探索里还遇到过中文文本在中间文件里显示乱码的问题，尤其是 logs\talkflower_zh\2-name2text.txt。这个问题提醒我们：训练流水线里不只要检查音频，还要检查文本编码、list 格式和 GPT-SoVITS 每个中间文件是否真的可读。

阶段 2：v1 初版模型先证明流程跑通

第一版实验名是 talkflower_zh。它使用较早的配置和初始清理方式训练，产出了：

类型 产物
SoVITS 权重 SoVITS_weights\talkflower_zh_e8_s1392.pth
GPT 权重 GPT_weights\talkflower_zh-e15.ckpt
测试音频 outputs\talkflower_zh_test_pcm16.wav

这个版本的意义是证明端到端流程已经通了：能训练，能加载权重，能生成 32 kHz mono PCM16 wav，音频也不是静音。但试听和 ASR 检查都显示它还只是“可用的初版”：内容有偏差，声音不够像，后续还出现了破音、音调偏高、声音发虚、吐字不清晰等问题。

这一步的结论很重要：训练流程跑通，不等于模型质量达标。 对碎片化语音来说，数据质量和筛选策略往往比继续盲目加轮数更关键。

阶段 3：数据审计解释了“为什么不像”

对 v1 数据做审计后，我们发现数据集虽然有 744 条，但平均长度只有约 1.53 秒，其中 147 条小于 1 秒，468 条在 1 到 2 秒之间。大量样本是“啊”“哎”“对”一类短促语气词，能提供的稳定音色和韵律信息很少。

观察 影响
大量音频少于 2 秒 对音色、节奏、长句稳定性帮助有限
语气词和短碎片过多 容易学到破碎的发声方式，而不是完整说话风格
个别异常文本或非中文片段 影响文本到语义 token 的对齐
某些目录更像特殊音效或 VoiceOnly 素材 会拉偏音色、音高和清晰度

于是优化方向从“继续训练 v1”转为“先清洗数据，再换 v2ProPlus 训练”。这也是整个探索路线里最关键的转折点。

阶段 4：从 clean 到 strict，逐步收紧训练集

我们把清理策略分成两个层级，而不是一次性把所有东西删掉。这样做的好处是能保留对比空间：如果 strict 过于保守，还能回到 clean；如果 clean 仍混入噪声，则可以进一步收紧。

列表 规则 输出
smbw_zh.list 原始中文 list，来自 CNzh / TWzh 的自动整理和标注 作为审计和回溯基线
smbw_zh_clean_v2pp.list 丢弃中文字符少于 4 个的片段，并默认排除文件名含 VoiceOnly 的素材 用于 v2ProPlus clean 实验
smbw_zh_strict_v2pp.list 在 clean 基础上继续排除文件名含 CommonBC 的素材 用于最终 v2pp strict 实验

这些规则来自实际试听反馈：忽略 TWzh\Voice\TalkFlower_VoiceOnly_Stream，忽略 CommonBC_*.mp3，只保留文本超过 3 个字的片段。它们不是抽象的数据清洗教条，而是围绕“声音是否更像、更稳、更清楚”不断验证出来的。

阶段 5：切换到 v2ProPlus，并用同一句话对比试听

清洗后我们切换到 GPT-SoVITS v2ProPlus。v2ProPlus 训练会产生额外中间特征，例如说话人向量相关文件，也对应 GPT_weights_v2ProPlus 和 SoVITS_weights_v2ProPlus 两组权重目录。 

.\scripts\stage-02-format-dataset.ps1 `
  -ListPath ".\data\smbw_zh_train\metadata\smbw_zh_strict_v2pp.list" `
  -AudioDir ".\data\smbw_zh_train\audio" `
  -ExpName "talkflower_zh_v2pp_strict" `
  -GpuNumbers "0"

.\scripts\stage-03-train-sovits.ps1 `
  -ExpName "talkflower_zh_v2pp_strict" `
  -Version v2ProPlus `
  -GpuNumbers "0" `
  -Epochs 10

.\scripts\stage-04-train-gpt.ps1 `
  -ExpName "talkflower_zh_v2pp_strict" `
  -Version v2ProPlus `
  -GpuNumbers "0" `
  -Epochs 18

在 clean 阶段，我们保留过一个试听效果比较好的样本：outputs\talkflower_zh_v2pp_ref05100_conservative.wav。它说明“数据清理 + v2ProPlus + 保守采样参数”的方向是对的。随后 strict 版本继续收紧素材，把它变成当前默认模型。

阶段 6：v2pp strict 成为当前默认版本

当前仓库默认的最新版模型是 talkflower_zh_v2pp_strict。默认推理配置使用：

项目 路径或内容
GPT 权重 GPT_weights_v2ProPlus\talkflower_zh_v2pp_strict-e18.ckpt
SoVITS 权重 SoVITS_weights_v2ProPlus\talkflower_zh_v2pp_strict_e10_s2120.pth
参考音频 data\smbw_zh_train\audio\TWzh__TalkFlower_Placement_Stream__Course_051_00.mp3
参考文本 只有被选中的人才能得到璀璨的光芒。
示例输出 outputs\talkflower_zh_v2pp_strict_ref05100_conservative.wav

这里的参考音频不是多余的。GPT-SoVITS 推理时，微调权重提供“这个角色如何说话”的能力，参考音频和参考文本则提供本次发声的条件，包括音色、韵律、音量、口吻和音频内容对齐。历史上第一次 smoke test 曾因为参考音频不足 3 秒失败，后来改用满足 3-10 秒要求的参考音频后，CUDA 推理成功生成测试 wav。

阶段 7：验证不只听声音，还要测速度和可部署性

训练完成后，我们验证了三类事情：能不能生成、生成效果是否稳定、推理速度是否适合使用。仓库里保留了推理脚本和测速脚本： 

.\scripts\infer-talkflower.ps1 `
  -Text "你好，我是闲聊花花。今天也一起开心地冒险吧。" `
  -Output ".\outputs\talkflower_latest.wav" `
  -Device cuda

.\scripts\benchmark-talkflower.ps1 -Warmup 1 -Runs 3

当前机器实测结果如下：

设备 平均耗时 平均生成音频时长 平均 RTF 结论
NVIDIA GeForce RTX 4070 / CUDA 2.717 秒 5.120 秒 0.531 快于实时播放
CPU 5.979 秒 5.000 秒 1.196 慢于实时播放

RTF 是 推理耗时 / 音频时长，小于 1 表示生成速度快于播放速度。这个结果说明，如果要构建 API service，应该让模型常驻 GPU，启动时加载 BERT、CNHuBERT、GPT 和 SoVITS，启动后先预热一次；不要每个请求重新启动 Python 或重新加载权重。

最终固化：把探索变成可复用脚本

探索结束后，最重要的不是只留下一个 wav，而是把每一步变成未来可以重跑的脚本：

01 数据准备

scripts\stage-01-prepare-data.ps1 解压中文语音、ASR 标注、生成 clean/strict list 和 dropped 报告。

02 格式化训练集

scripts\stage-02-format-dataset.ps1 生成文本、HuBERT、32 kHz wav、语义 token 和 v2ProPlus 需要的中间文件。

03 训练 SoVITS

scripts\stage-03-train-sovits.ps1 训练声学/声码器侧权重，输出到 SoVITS_weights_v2ProPlus。

04 训练 GPT

scripts\stage-04-train-gpt.ps1 训练文本到语义 token 生成能力，输出到 GPT_weights_v2ProPlus。

05 测试模型

scripts\stage-05-test-model.ps1 用固定参考音频和固定文本生成 wav，便于跨实验对比。

如果要完整重跑，可以使用： 

.\scripts\train-all.ps1 `
  -ZipPath "C:\Users\richard\Downloads\smbw_voice.zip" `
  -ExpName "talkflower_zh_v2pp_strict" `
  -GpuNumbers "0"

如果已有数据和中间特征，只想继续训练，则可以跳过前两步： 

.\scripts\train-all.ps1 `
  -ExpName "talkflower_zh_v2pp_strict" `
  -GpuNumbers "0" `
  -SkipData `
  -SkipFormat

这次探索得到的经验

  1. 先跑通，再追求像。 v1 的价值是验证流程，而不是最终质量。
  2. 碎片化语音必须审计。 样本数量多不代表有效信息多，过短语气词会稀释音色学习。
  3. 清理规则要能回溯。 clean 和 strict 分层保留了实验对比空间，也让每条丢弃规则有据可查。
  4. 固定参考音频和测试句。 只有控制变量，才能判断 v1、v2ProPlus clean、v2pp strict 哪个真的更好。
  5. 验证包括速度。 对 TTS 来说，能生成 wav 只是第一步；RTF、冷启动、模型常驻方式决定它能不能服务化。

最终，talkflower_zh_v2pp_strict 成为当前默认版本。它不是一次命令训练出来的，而是经历了 v1 跑通、问题试听、数据审计、clean 清洗、v2ProPlus 迁移、strict 收紧和推理测速之后，逐步收敛出来的工程结果。
